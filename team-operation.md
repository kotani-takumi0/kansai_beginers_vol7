# チーム運用方針

## チーム構成

| メンバー | スキルレベル | 開発スタイル |
|----------|-------------|-------------|
| 自分 | バイブコーディング（Claude Code活用フルスタック） | 設計 + 実装 |
| 相方 | 完全初心者 | Claude Code活用で実装に挑戦 |

## 基本方針

**画面単位で分担し、それぞれが独立して開発 → マージする**

### なぜこの方針か

- AIの進歩により、初心者でも画面単位の実装が可能になった
- Git分担・マージの経験は初学者にとって大きな成長機会
- 審査基準に「チームワークと学習意欲」が明記されており、加点が見込める
- 実装自体はClaude Codeで効率的にできるため、詰まるリスクが低い

## 役割分担

### 自分の担当

- プロジェクトの初期セットアップ
- API連携・データ受け渡しなどの複雑なロジック
- 体験型入力のある画面（質問画面等）
- 相方への仕様説明・サポート

### 相方の担当

- 結果表示系の画面（カード表示・シェア等）
- デザイン・アニメーション
- ユーザーテスト（最初のユーザーとしてフィードバック）
- プレゼン資料の作成

### 共通

- 困ったらClaude Codeに聞く
- マージ作業は必ず2人で一緒にやる（学びのポイント）
- お互いの画面を説明し合う時間を設ける

## 開発フロー全体像

### Kiro Spec → Issue → 開発サイクル

```
[Phase 0] Kiro Spec-Driven Development
  requirements.md（何を作るか）
      ↓ 2人でレビュー
  design.md（どう作るか）
      ↓ 2人でレビュー
  tasks.md（具体的な作業リスト）
      ↓ 担当を割り振り

[Phase 1] Claude CodeでIssueを一括作成
      ↓ GitHub上で2人でレビュー

[Phase 2] 開発サイクル（繰り返し）
  Issue確認 → ブランチ作成 → 開発 → Draft PR → 説明し合い → マージ
                                                  ↑
                                            ここが学びの核
```

### Kiroが生成するファイル

| フェーズ | ファイル | 内容 |
|---------|---------|------|
| Requirements | `requirements.md` | ユーザーストーリー + 受け入れ基準 |
| Design | `design.md` | 技術設計・画面構成・データフロー |
| Tasks | `tasks.md` | 順序付き・依存関係付きの実装タスク一覧 |

Kiroが「何を・どう作るか」を整理 → Issueの定義で迷わなくなる。

### Issue作成はClaude Codeに任せる

```
あなた：「tasks.mdをもとにIssueを作って」とClaude Codeに指示
  ↓
Claude Code：gh コマンドでIssueを一括作成
  ↓
2人：GitHub上で確認・修正・担当割り振り
  ↓
各自：担当Issueを見て作業開始
```

## 開発サイクル詳細

### 1. Issueで作業を管理する

コードを書く前に、必ずIssueを立てる。

```
例：
Issue #1: プロジェクト初期セットアップ（自分）
Issue #2: 質問画面 - スライダーUI（自分）
Issue #3: 結果カード画面 - レイアウト（相方）
Issue #4: 結果カード画面 - シェアボタン（相方）
```

なぜやるか：
- 「今何をやってるか」がお互い見える
- 相方が「次何すればいい？」と迷わない
- 完了したIssueが積み上がってモチベーションになる

#### Issueの書き方

```markdown
タイトル: 何をするか（短く）
---
## 概要
なぜこれをやるのか（1〜2行）

## やること
- [ ] 具体的なタスク1
- [ ] 具体的なタスク2
- [ ] 具体的なタスク3

## 完了条件
これができたらOK、という基準

## 担当
誰がやるか
```

#### 相方向けIssueにはヒント欄を追加

```markdown
## ヒント
- ○○のデザインは src/components/common/Button.tsx を参考に
- △△には □□ というライブラリを使う
- わからなかったらClaude Codeに「〜〜して」と聞いてみて
```

#### Issueの粒度

```
大きすぎ → 「質問画面を全部作る」（何日かかるか不明）
ちょうどいい → 「スライダーUIを作る」（2〜4時間で終わる）
小さすぎ → 「ボタンの色を変える」（Issueにするほどでもない）
```

#### ラベルで分類

| ラベル | 意味 |
|--------|------|
| `画面:質問` | 質問画面に関するIssue |
| `画面:結果` | 結果画面に関するIssue |
| `共通` | 共通部品・設定 |
| `優先:高` | 先にやるべき |
| `優先:低` | 余裕があれば |

### 2. ブランチの切り方

```bash
# Issueごとに1ブランチ（短命が鉄則）
git checkout -b feature/#2-question-slider

# 命名規則
feature/#Issue番号-短い説明
```

1ブランチ = 1つの小さな機能。「質問画面全部」ではなく「スライダーUI」くらいの粒度。

理由：AIが大量のコードを生成するので、大きなブランチだとレビューが追いつかない。

### 3. コミットの粒度

```bash
# NG: まとめてドーン
git commit -m "質問画面作った"

# OK: 論理単位で分割
git commit -m "feat: スライダーコンポーネントの骨組み作成"
git commit -m "style: スライダーの色とサイズを調整"
git commit -m "feat: スライダーの値をstateに保存"
```

Claude Codeで一気に生成しても、コミットは手動で分ける。これが相方への説明材料になる。

### 4. Draft PRを即座に開く

```bash
# ブランチを作ったらすぐpush
git push -u origin feature/#2-question-slider

# GitHubでDraft PRを作成
gh pr create --draft --title "feat: 質問画面スライダー" --body "Issue #2 対応"
```

Draft PRの効果：
- 相方が「今こんなの作ってるんだ」とリアルタイムで見れる
- 作業中でも気軽にコメントし合える
- 完成を待たずにフィードバックをもらえる

### 5. 説明し合い（= コードレビュー）

ハッカソンで正式なコードレビューは重すぎる。代わりに：

**タイミング：Draft PR → Ready にする前に対面で説明**

自分 → 相方に説明する場合：
> 「このファイルでスライダーの見た目を作ってて、
> ここでユーザーの入力値をstateに保存してて、
> この値が最終的に結果画面に渡される」

相方 → 自分に説明する場合：
> 「結果カードのレイアウトをこのコンポーネントで作った。
> AIに色の調整を頼んだらこうなった。
> ここの部分がまだわからない」→ 一緒に理解する

**「わからない部分」を共有するのが一番の学び**。全部理解する必要はない。

### 6. マージの手順

```bash
# PRをReadyにする
gh pr ready

# mainにマージ（GitHub上でSquash Mergeを選択）
# → 細かいコミットが1つにまとまる

# マージ後、ローカルを更新
git checkout main
git pull origin main

# 使い終わったブランチを削除
git branch -d feature/#2-question-slider
```

Squash Merge推奨。AI生成の細かいコミットがmainを汚さない。

## ファイル分離によるコンフリクト防止

```
src/
├── components/
│   ├── questions/    ← 自分だけが触る
│   │   ├── QuestionPage.tsx
│   │   └── Slider.tsx
│   ├── results/      ← 相方だけが触る
│   │   ├── ResultCard.tsx
│   │   └── ShareButton.tsx
│   └── common/       ← 自分が管理、相方は触らない
│       ├── Layout.tsx
│       └── theme.ts
```

**同じファイルを同時に編集しない** — これが最大のコンフリクト防止策。

共通部分の変更ルール：
- `common/` は自分が管理する
- 相方が共通部分を変えたい場合は口頭で伝えてもらう
- 自分が変更してpush → 相方がpullで取り込む

## Git運用ルール

### ブランチ戦略

```
main（直接触らない）
├── feature/#2-question-slider  ← 自分
└── feature/#3-result-card      ← 相方
```

### 相方が覚えるべきコマンド（最低限）

| やりたいこと | コマンド |
|-------------|---------|
| 変更を保存 | `git add . && git commit -m "メッセージ"` |
| アップロード | `git push` |
| 最新を取り込む | `git pull origin main` |
| 今の状態を確認 | `git status` |

### やってはいけないこと

- `main` ブランチで直接作業しない
- `git push --force` は絶対しない
- 困ったらコマンドを打つ前に聞く

## CLAUDE.mdでルール共有

プロジェクトルートに配置。Claude Codeが自動で読み込む。

```markdown
# CLAUDE.md

## プロジェクト概要
地元紹介名刺アプリ（ハッカソン）

## 技術スタック
（決まったら記載）

## ディレクトリルール
- src/components/questions/ → 自分の担当
- src/components/results/  → 相方の担当
- src/components/common/   → 自分が管理

## コミットルール
- Conventional Commits形式（feat/fix/style）
- 日本語OK

## 触ってはいけないファイル
- 相手の担当ディレクトリは直接編集しない
```

これがあると相方がClaude Codeを使うとき、自動的にルールに従ったコードが生成される。

## 開発開始前に2人で決めること

1. **デザイントークン** — 使用する色（2〜3色）、フォント、余白の基準
2. **データ形式** — 画面間で受け渡すデータのJSON構造
3. **共通レイアウト** — ヘッダー・フッター等の共通部分
4. **技術スタック** — フレームワーク・ライブラリの選定

## 1日のタイムライン例

```
09:00  一緒にIssueを確認、今日のタスクを決める
09:15  各自ブランチを切ってDraft PR作成
09:20  各自開発（Claude Codeで実装）
11:00  説明し合いタイム（15分）
        → フィードバック、方向修正
11:15  各自開発続行
13:00  マージタイム（一緒にやる）
        → コンフリクトがあれば一緒に解決
13:30  動作確認、次のIssueへ
```
