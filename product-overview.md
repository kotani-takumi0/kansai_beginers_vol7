# プロダクト概要

## コンセプト

**地元紹介 × 名刺アプリ — 目の前の人との会話を生む装置**

すれ違い通信・ポケモン交換のワクワク感を、地元紹介で再現する。

## ユースケース

地方出身者が新生活で新しい土地に行き、そこでできた友達に「あなたの地元はどんなところなの？」と聞かれた際に使えるアプリ。会話が広がるアプリ。

### 想定シーン

```
友達：「地元どこなん？」
自分：「大阪やで、ちょっとこれ見て」→ アプリを見せる
友達：「え、なにこれ面白い、俺もやる！」
→ お互いの名刺を作る
→ スマホをぶつけて交換
→ 交換演出が流れる（ポケモン交換のような演出）
→ 相手の名刺が届く
→ 一致/不一致を見て会話が広がる
→ 仲が深まる
```

## コア体験

### 1. 名刺を作る

- 出身地を入力する
- AIが「その地域特有の、本人は普通だと思っているけど議論になるネタ」を生成する
- ユーザーが自分の立場を選ぶ（同意する/しない）
- それが「地元名刺」として完成する

### 2. 名刺を交換する — すれ違い通信の現代版

- **スマホ同士をぶつけて交換**する
- 技術的にはDeviceMotion API（加速度センサー）で動作検知 + WebSocketで同期
- Webブラウザだけで完結。ネイティブアプリ不要
- ぶつけた瞬間 → 交換演出 → 相手の名刺が届く

#### 交換演出（ポケモン交換のイメージ）

```
ぶつける
  → 画面が反応（振動・光・音）
  → 通信中の演出（ドキドキの「間」を作る）
  → 相手の名刺が画面に飛んでくる
  → 「おぉ！」
```

この**ぶつけてから結果が出るまでの演出**がアプリの勝負所。
ポケモン交換で大事だったのは技術ではなく「間」と「演出」だった。

### 3. 比較して会話が生まれる

- 2人の名刺が並び、同じ項目を選んでいたら「一致」として可視化される
- 一致 → 「お前もそうなん！」→ 共感で盛り上がる
- 不一致 → 「え、マジで！？」→ 議論で盛り上がる
- どちらに転んでも会話が広がる

## アハモーメント（4段階）

| タイミング | 驚き |
|-----------|------|
| ネタ生成時 | 「え、これ普通じゃなかったの？」 |
| ぶつけた瞬間 | 物理的な動作 + 画面の反応 → 「おぉ！」 |
| 交換演出中 | ポケモン交換的なドキドキ感 |
| 比較結果表示 | 一致/不一致が見えて「え！」→ 会話スタート |

## 課題：新生活の自己紹介はSNSに依存している

### 原体験

自分たちが実際に関西から遠方に行った。新しい環境で友達を作るとき、当たり前のようにインスタを交換する。投稿やストーリー、アイコンから共通点を探して会話を広げていた。

### 既存アプリの「痛み」

| アプリ | やっていること | 痛み |
|--------|-------------|------|
| インスタ交換 | 投稿・ストーリーから共通点を探す | **コンテンツがない人は会話が広がらない。SNSを頑張っている人だけが得をする** |
| 名刺アプリ（Lit.Linkなど） | 連絡先・SNSリンクをまとめて交換 | **情報を渡すだけ。「フォローしたよ」で会話が終わる** |
| マッチングアプリ | プロフィールで相性を判定 | **初対面の対面シーンには使えない。リアルの場の会話を助けてくれない** |

**共通する痛み：どのアプリも「目の前の人との会話を広げる」ことを目的にしていない。**

### このアプリが解決すること

```
痛み：SNSのコンテンツ量がないと会話が広がらない
  ↓
解決：出身地さえあれば、会話のきっかけが自動で生まれる
```

- 投稿ゼロでも関係ない。出身地は誰にでもある
- 連絡先を交換するのではなく、**会話のきっかけを交換する**
- 一致/不一致がその場で見えるから、**交換した瞬間に会話が始まる**

**「SNSを頑張っていない人でも、初対面の会話を楽しめる」。これが既存アプリにはない価値。**

## なぜこのアプリが新しいのか

従来の体験はすべてバラバラに存在していた。

| 既存のもの | 限界 |
|-----------|------|
| 県民性ネタ（テレビ） | 一方通行。見て「へー」で終わり |
| あるある診断（Web） | 1人遊び。共有しても「わかる〜」で終わり |
| 名刺交換アプリ | 情報交換で終わり。会話は生まれない |
| インスタ交換 | コンテンツがない人は会話が広がらない |
| すれ違い通信 | 専用ハードが必要。もう体験できない |

このアプリは、これらを**一体化**することで「2人の間に会話が生まれる」という、どれか単体では成立しない体験を作る。

### 技術的な独自性

**「スマホをぶつけて交換」をWebブラウザだけで実現する。**

- DeviceMotion API — 加速度センサーでぶつける動作を検知
- WebSocket — 同タイミングで動作検知された2台をマッチング
- ネイティブアプリなしで、すれ違い通信的な体験をWebで再現

## プロダクトの目的

**SNSでバズることでも、1人で楽しむことでもない。目の前の人との関係を深めること。**

- 対面で「これやってみてよ」と勧めたくなる
- 別の人と交換するたびに違う発見がある
- 新生活シーズンのタイミングと相性が良い

## 審査基準との整合

| 審査項目 | 強み |
|---------|------|
| アハモーメント(25%) | 4段階の驚き。ぶつけて交換の物理的体験 + 比較結果の発見 |
| 技術的完成度(20%) | DeviceMotion API + WebSocketの技術的挑戦。デザイン・UI実装の品質 |
| UI/UX(20%) | ぶつける→交換演出→比較表示の体験設計。名刺メタファーの直感性 |
| 革新性(20%) | すれ違い通信の現代版をWebで再現。既存要素の一体化 |
| プレゼン(15%) | 審査員2人にその場でぶつけて交換してもらう体験型デモ |

## MVP範囲

### Must — Day 1（これがないと成立しない）

1. 出身地入力 → AIがネタ生成 → 立場を選ぶ → 名刺完成
2. 名刺をURL/QRで共有して交換
3. 2人の名刺を並べて比較表示（一致/不一致）

**これだけでコア体験「会話が広がる」は成立する。**
2人で画面分担して確実に完成させる。

### Should — Day 2（技術的挑戦）

4. スマホをぶつけて交換（DeviceMotion + WebSocket）
5. 交換演出（ポケモン交換的なアニメーション）

**安定性 + 技術的挑戦で技術点を狙う。**
機能の数より、ぶつけて交換を完成させる方がインパクトが大きい。

### Could（余裕があれば）

6. 名刺デザインのカスタマイズ
7. 交換履歴の保存

### 捨てる

- ログイン/アカウント機能（普通すぎて技術点にならない）
- データベースへの永続保存
- ネイティブアプリ（Webで全機能実現できる）
