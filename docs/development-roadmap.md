## クリティカルパス

**プロジェクト全体の最短経路（これが遅れると全体が遅れる）:**

```
#1 プロジェクト初期設定
 ↓
#3 型定義  ──→  #6 AI API
 ↓                 ↓
#4 Encoder     #8 ネタ生成画面（相方）
 ↓                 ↓
#10 共有画面     #9 名刺カード（相方）
     ↓               ↓
     └──── #11 受信画面（相方）
                ↓
           #12 比較表示（相方）
```

### ボトルネックになりやすい箇所

| リスク | 理由 | 対策 |
|--------|------|------|
| **#3 型定義が遅れる** | 全コンポーネントが依存 | Day 1 午前中に必ず完了 |
| **#6 AI APIが遅れる** | #8 ネタ生成画面がブロックされる | プロンプト設計を事前に済ませる。モックAPIを先に用意 |
| **相方の #8 が遅れる** | #9 → #10 → #11 → #12 が全て詰まる | 画面単位で動くものを先に作り、API繋ぎ込みは後 |
| **#13 Socket.IOサーバー** | Day 2のぶつけ交換が全てブロック | Day 1 夜に先行着手も検討 |

## 判断ポイント

| タイミング | 判断内容 |
|-----------|---------|
| **Day 1 午後** | AI APIが動いているか？ → 動かなければモックデータで相方をブロック解除 |
| **Day 1 夜** | Must機能が全て動くか？ → 動かなければDay 2午前もMust対応に充てる |
| **Day 2 午前** | ぶつけ交換に着手できるか？ → 無理ならURL/QR交換の磨き込みに切り替え |
| **Day 2 午後** | 交換演出のクオリティ → 時間がなければシンプルな演出で妥協 |
| **Day 2 夜** | 全フロー結合完了か？ → 未完なら交換演出を削ってDay 3で結合に集中 |

## 最悪シナリオのフォールバック

```
ぶつけ交換が間に合わない場合:
  → URL/QR交換だけでもコア体験は成立する
  → 審査時は「技術的にここまで実装した」とデモ可能

AI APIが不安定な場合:
  → 固定の議論ネタデータセット（10県分）を用意しておく
  → API復旧次第切り替え
```
